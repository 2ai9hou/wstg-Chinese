# 测试ORM注入

## 概述

[对象关系映射（ORM）注入](https://capec.mitre.org/data/definitions/109.html)是一种使用SQL注入攻击ORM生成的数据访问对象模型的攻击。从测试人员的角度来看，此攻击与SQL注入攻击几乎相同。但是，注入漏洞存在于ORM层生成的代码中。

使用ORM工具的好处包括快速生成与关系数据库通信的对象层、标准化这些对象的代码模板，以及它们通常提供一组安全函数来防止SQL注入攻击。ORM生成的对象可以使用SQL或在某些情况下使用SQL变体来对数据库执行CRUD（创建、读取、更新、删除）操作。但是，如果Web应用程序使用ORM生成的对象的方法可以接受未清理的输入参数，则可能容易受到SQL注入攻击。

## 如何测试

ORM层可能容易受到漏洞的影响，因为它们扩展了攻击面。与其直接使用SQL查询攻击应用程序，不如专注于滥用ORM层发送恶意SQL查询。

### 识别ORM层

为了有效地测试和理解请求和后端查询之间发生的情况，并且与进行适当测试的所有相关一样，识别手头应用程序使用的技术至关重要。通过遵循[信息收集](../01-Information_Gathering/README.md)章节，您应该了解正在使用的技术。检查此[语言与其相应ORM的映射列表](https://en.wikipedia.org/wiki/List_of_object-relational_mapping_software)。

### 滥用ORM层

在识别出可能使用的ORM后，必须了解其解析器的功能，并研究滥用它或应用程序是否使用旧版本的方法，从而识别与其使用的库相关的CVE。有时，ORM层没有正确实现，因此允许测试人员进行正常的[SQL注入](05-Testing_for_SQL_Injection.md)，而不必担心ORM层。

#### 弱ORM实现

一种易受攻击的场景是ORM层实现不当，摘自[SANS](https://software-security.sans.org/developer-how-to/fix-sql-injection-in-java-hibernate)：

```java
List results = session.createQuery("from Orders as orders where orders.id = " + currentOrder.getId()).list();
List results = session.createSQLQuery("Select * from Books where author = " + book.getAuthor()).list();
```

上述没有实现位置参数，这允许开发人员将输入替换为`?`。例如：

```java
Query hqlQuery = session.createQuery("from Orders as orders where orders.id = ?");
List results = hqlQuery.setString(0, "123-ADB-567-QTWYTFDL").list(); // 0是第一个位置，其中动态替换为设置的字符串
```

此实现将验证和清理留给ORM层，绕过它的唯一方法是识别ORM层的问题。

#### 易受攻击的ORM层

ORM层是代码，大多数时候是第三方库。它们可能像任何其他代码一样容易受到攻击。一个例子是[sequelize ORM npm库](https://snyk.io/blog/sequelize-orm-npm-library-found-vulnerable-to-sql-injection-attacks/)，该库在2019年发现存在漏洞。在[ RIPS Tech](https://www.ripstech.com/)进行的另一项研究中，发现了[Java使用的Hibernate ORM](https://hibernate.org/orm/)中的绕过。

根据他们的[博客文章](https://blog.ripstech.com/2020/exploiting-hibernate-injections/)，可以列出以下测试人员识别问题的备忘单：

| DBMS       | SQL注入                                                         |
|------------|-----------------------------------------------------------------------|
| MySQL      | `abc\' INTO OUTFILE --`                                               |
| PostgreSQL | `$$='$$=chr(61) || chr(0x27) and 1=pg_sleep(2) || version()'`         |
| Oracle     | `NVL(TO_CHAR(DBMS_XMLGEN.getxml('select 1 where 1337>1')),'1')!='1'`  |
| MS SQL     | `1<LEN(%C2%A0(select%C2%A0top%C2%A01%C2%A0name%C2%A0from%C2%A0users)` |

另一个例子是[Laravel Query-Builder](https://laravel.com/docs/7.x/queries)，在2019年发现[容易受到攻击](https://freek.dev/1317-an-important-security-release-for-laravel-query-builder)。
