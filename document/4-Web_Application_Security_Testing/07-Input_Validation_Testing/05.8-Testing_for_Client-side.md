# 测试客户端

## 概述

当应用程序实现[Web SQL数据库](https://www.w3.org/TR/webdatabase/)技术且不正确验证输入或参数化其查询变量时，会发生客户端SQL注入。使用JavaScript（JS）API调用（如`openDatabase()`）来操纵此数据库，该调用创建或打开现有数据库。

## 测试目标

以下测试场景将验证是否进行了适当的输入验证。如果实现存在漏洞，攻击者可以读取、修改或删除存储在数据库中的信息。

## 如何测试

### 识别Web SQL DB的使用

如果被测应用程序实现了Web SQL DB，以下三个调用将在客户端核心中使用：

- `openDatabase()`
- `transaction()`
- `executeSQL()`

以下代码显示API的实现示例：

```javascript
var db = openDatabase(shortName, version, displayName, maxSize);

db.transaction(function(transaction) {
    transaction.executeSql('INSERT INTO LOGS (time, id, log) VALUES (?, ?, ?)', [dateTime, id, log]);
});
```

### Web SQL DB注入

确认`executeSQL()`的使用后，攻击者已准备好测试和验证其实现的安全性。

Web SQL DB的实现基于[SQLite的语法](https://www.sqlite.org/lang.html)。

#### 绕过条件

以下示例显示如何在客户端利用此漏洞：

```javascript
// URL示例：https://example.com/user#15
var userId = document.location.hash.substring(1,); // 获取ID而不带hash -> 15

db.transaction(function(transaction){
    transaction.executeSQL('SELECT * FROM users WHERE user = ' + userId);
});
```

要返回所有用户的信息（而不仅仅是与攻击者对应的用户），可以使用URL片段中的`15 OR 1=1`。

> **_注意：_**  在SQL查询中注入条件OR 1=1时要小心。虽然在您注入的初始上下文中这可能是无害的，但应用程序通常在多个不同的查询中使用单个请求中的数据。例如，如果您的条件到达UPDATE或DELETE语句，可能导致意外的数据丢失。

有关其他SQL注入payload，请转到[测试SQL注入](05-Testing_for_SQL_Injection.md)场景。
