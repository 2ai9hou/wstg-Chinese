# 测试MS Access

## 概述

如通用[SQL注入](https://owasp.org/www-community/attacks/SQL_Injection)部分所述，SQL注入漏洞发生在用户提供的输入用于构造SQL查询而没有充分约束或清理时。这类漏洞允许攻击者在用于连接数据库的用户权限下执行SQL代码。在本节中，将讨论利用[Microsoft Access](https://en.wikipedia.org/wiki/Microsoft_Access)特定功能的SQL注入技术。

## 如何测试

### 指纹识别

在对SQL驱动的应用程序进行测试时，对特定数据库技术进行指纹识别是正确评估潜在漏洞的第一步。一种常见方法是在其中注入标准SQL注入攻击模式（例如单引号、双引号……），以触发数据库异常。假设应用程序不使用自定义页面处理异常，则可以通过观察错误消息对底部DBMS进行指纹识别。

根据使用的特定Web技术，MS Access驱动的应用程序将以以下错误之一进行响应：

`Fatal error: Uncaught exception 'com_exception' with message Source: Microsoft JET Database Engine`

或者

`Microsoft JET Database Engine error '80040e14'`

或者

`Microsoft Office Access Database Engine`

在所有情况下，我们都确认我们正在测试使用MS Access数据库的应用程序。

### 基本测试

不幸的是，MS Access不支持传统上用于SQL注入测试的常见运算符，包括：

- 没有注释字符
- 没有堆叠查询
- 没有LIMIT运算符
- 没有SLEEP或BENCHMARK类似的运算符
- 还有许多其他

但是，可以通过组合多个运算符或使用替代技术来模拟这些功能。如前所述，无法使用插入字符`/*`、`--`或`#`的技巧来截断查询。但是，我们可以通过注入'null'字符来绕过此限制。在SQL查询中使用空字节`%00`会导致MS Access忽略所有剩余字符。这可以解释为在数据库使用的内部表示中所有字符串都是NULL终止的。值得指出的是，`null`字符有时可能会在Web服务器级别截断字符串。在这些情况下，我们也可以使用另一个字符：`0x16`（URL编码格式为`%16`）。

考虑以下查询：

`SELECT [username],[password] FROM users WHERE [username]='$myUsername' AND [password]='$myPassword'`

我们可以用以下两个URL截断查询：

- `https://www.example.com/page.asp?user=admin'%00&pass=foo`
- `https://www.example.com/page.app?user=admin'%16&pass=foo`

MS Access不支持`LIMIT`运算符，但是可以使用`TOP`或`LAST`运算符来限制结果数量。

`https://www.example.com/page.app?id=2'+UNION+SELECT+TOP+3+name+FROM+appsTable%00`

通过组合这两个运算符，可以选择特定结果。可以使用`& (%26)`和`+ (%2b)`字符进行字符串连接。

还有许多其他可在测试SQL注入时使用的函数，包括但不限于：

- ASC：获取作为输入传递的字符的ASCII值
- CHR：获取作为输入传递的ASCII值的字符
- LEN：返回作为参数传递的字符串的长度
- IIF：这是IF构造，例如以下语句`IIF(1=1, 'a', 'b')`返回`a`
- MID：此函数允许您提取子字符串，例如以下语句`mid('abc',1,1)`返回`a`
- TOP：此函数允许您指定查询应从顶部返回的最大结果数。例如，`TOP 1`将只返回1行。
- LAST：此函数用于仅选择一组行的最后一行。例如，以下查询`SELECT last(*)` FROM users将返回结果的倒数第二行。

其中一些运算符对于利用盲SQL注入至关重要。对于其他高级运算符，请参阅参考资料中的文档。

#### 属性枚举

为了枚举数据库表的列，可以使用常见的基于错误的技术。简言之，我们可以通过分析错误消息并使用不同的选择器重复查询来获取属性名称。例如，假设我们知道一列的存在，我们也可以使用以下查询获取其余属性的名称：

`' GROUP BY Id%00`

在收到的错误消息中，可以观察到下一列的名称。此时，我们可以迭代该方法直到获得所有属性的名称。如果我们不知道第一个属性的名称，我们仍然可以插入一个虚构的列名并在错误消息中获取第一个属性的名称。

#### 获取数据库架构

MS Access中存在多个默认系统表，可能用于获取表名和列。不幸的是，在最近发布的MS Access数据库的默认配置中，这些表不可访问。不过，总是值得尝试：

- MSysObjects
- MSysACEs
- MSysAccessXML

例如，如果存在联合SQL注入漏洞，可以使用以下查询：

`' UNION SELECT Name FROM MSysObjects WHERE Type = 1%00`

或者，始终可以使用标准词汇表（例如[FuzzDb](https://github.com/fuzzdb-project/fuzzdb)）暴力破解数据库架构。

在某些情况下，开发人员或系统管理员没有意识到在应用程序webroot中包含实际的`.mdb`文件可能允许下载整个数据库。可以使用以下查询推断数据库文件名：

`https://www.example.com/page.app?id=1'+UNION+SELECT+1+FROM+name.table%00`

其中`name`是`.mdb`文件名，`table`是一个有效的数据库表。对于受密码保护的数据库，可以使用多种软件实用程序来破解密码。请参阅参考资料。

### 盲SQL注入测试

[盲SQL注入](https://owasp.org/www-community/attacks/Blind_SQL_Injection)漏洞绝不是测试真实应用程序时最容易利用的SQL注入。在最新版本的MS Access中，执行shell命令或读取/写入任意文件也不可行。

在盲SQL注入的情况下，攻击者只能通过评估时间差异或应用程序响应来推断查询结果。假设读者已经了解盲SQL注入攻击的理论，因为本节的其余部分将专注于MS Access特定细节。

使用以下示例：

`https://www.example.com/index.php?myId=[sql]`

其中ID参数用于以下查询：

`SELECT * FROM orders WHERE [id]=$myId`

让我们考虑`myId`参数容易受到盲SQL注入。作为攻击者，我们想要提取`users`表中`username`列的内容，假设我们已经披露了数据库架构。

可用于推断第10行用户名的第一个字符的典型查询：

`https://www.example.com/index.php?id=IIF((select%20MID(LAST(username),1,1)%20from%20(select%20TOP%2010%20username%20from%20users)='a',0,'no')`

如果第一个字符是`a`，查询将返回`0`，否则返回字符串`no`。

通过组合`IFF, MID, LAST`和`TOP`函数，可以提取特定选定行的用户名的第一个字符。由于内部查询返回一组记录，而不仅仅是一条，因此不能直接使用它。幸运的是，我们可以组合多个函数来提取特定字符串。

假设我们想要检索第10行的用户名。首先，我们可以使用以下查询使用TOP函数选择前10行：

`SELECT TOP 10 username FROM users`

然后，使用此子集，我们可以使用LAST函数提取最后一行。一旦我们只有一行，并且恰好是包含我们字符串的那一行，我们可以使用IFF、MID和LAST函数来推断用户名的实际值。在我们的示例中，我们使用IFF返回数字或字符串。使用此技巧，我们可以通过观察应用程序错误响应来区分我们是否有真实响应。因为`id`是数字，所以与字符串的比较会导致SQL错误，可能通过`500 Internal Server Error`页面泄露。否则，可能会返回标准的`200 OK`页面。

例如，我们可以使用以下查询：

`https://www.example.com/index.php?id='%20AND%201=0%20OR%20'a'=IIF((select%20MID(LAST(username),1,1)%20from%20(select%20TOP%2010%20username%20from%20users))='a','a','b')%00`

如果第一个字符是'a'，则为真，否则为假。

如前所述，此方法允许推断数据库中的任意字符串值：

1. 尝试所有可打印值，直到找到匹配项
2. 使用`LEN`函数推断字符串的长度，或者在我们找到所有字符后简单地停止

也可以通过滥用[重量查询](https://docs.microsoft.com/en-us/previous-versions/tn-archive/cc512676(v=technet.10))来进行基于时间的盲SQL注入。
