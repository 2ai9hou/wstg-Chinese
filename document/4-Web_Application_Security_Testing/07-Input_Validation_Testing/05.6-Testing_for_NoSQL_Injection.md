# 测试NoSQL注入

## 概述

NoSQL数据库比传统SQL数据库提供更宽松的一致性限制。通过需要更少的关系约束和一致性检查，NoSQL数据库通常提供性能和扩展优势。然而，这些数据库仍然可能容易受到注入攻击，即使它们不使用传统SQL语法。因为这些NoSQL注入攻击可以在[过程语言](https://en.wikipedia.org/wiki/Procedural_programming)中执行，而不是在[声明性SQL语言](https://en.wikipedia.org/wiki/Declarative_programming)中执行，所以潜在影响比传统SQL注入更大。

NoSQL数据库调用是用应用程序的编程语言、自定义API调用或根据通用约定（如`XML`、`JSON`、`LINQ`等）格式编写的。恶意输入可能不会触发主要应用程序清理检查。例如，过滤常见HTML特殊字符（如`< > & ;`）不会防止针对JSON API的攻击，其中特殊字符包括`/ { } :`。

有数百种NoSQL数据库可用于应用程序，提供各种语言和关系模型的API。每个都提供不同的功能和限制。因为它们之间没有通用语言，所以示例注入代码不会适用于所有NoSQL数据库。因此，任何测试NoSQL注入攻击的人员都需要熟悉语法、数据模型和底层编程语言才能制作特定测试。

NoSQL注入攻击可能在应用程序的不同区域执行，而不是传统SQL注入。在SQL注入将在数据库引擎内执行的地方，NoSQL变体可能在应用程序层或数据库层执行，这取决于所使用的NoSQL API和数据模型。通常，NoSQL注入攻击将在攻击字符串被解析、评估或连接到NoSQL API调用的位置执行。

附加时序攻击可能与NoSQL数据库中缺乏并发检查相关。这些不在注入测试范围内。在编写时，MongoDB是使用最广泛的NoSQL数据库，因此所有示例都将使用MongoDB API。

## 如何测试

### 测试MongoDB中的NoSQL注入漏洞

MongoDB API期望BSON（二进制JSON）调用，并包括安全的BSON查询组装工具。但是，根据MongoDB文档——未序列化的JSON和[JavaScript表达式](https://docs.mongodb.org/manual/faq/developers/#javascript)在多个替代查询参数中允许。以下是最常用允许任意JavaScript输入的API调用`$where`运算符。

MongoDB `$where`运算符通常用作简单过滤器或检查，因为它在SQL中。

`db.myCollection.find( { $where: "this.credits == this.debits" } );`

或者，JavaScript也可用于评估以允许更高级的条件。

`db.myCollection.find( { $where: function() { return obj.credits - obj.debits < 0; } } );`

### 示例1

如果攻击者能够操纵传递到`$where`运算符的数据，该攻击者可以包含任意JavaScript作为MongoDB查询的一部分。如果用户输入直接传递到MongoDB查询而没有清理的代码中存在漏洞，则会暴露该漏洞。

`db.myCollection.find( { active: true, $where: function() { return obj.credits - obj.debits < $userInput; } } );`

与测试其他类型的注入一样，不需要完全利用漏洞来演示问题。通过注入与目标API语言相关的特殊字符并观察结果，测试人员可以确定应用程序是否正确清理了输入。例如，在MongoDB中，如果传递包含以下任何特殊字符的字符串而未进行清理，将触发数据库错误。

`' " \ ; { }`

使用普通SQL注入，类似漏洞将允许攻击者执行任意SQL命令——随心所欲地公开或操作数据。但是，因为JavaScript是一种功能齐全的语言，不仅允许攻击者操作数据，还可以运行任意代码。例如，不是仅仅在测试时导致错误，完整漏洞利用将使用特殊字符来构造有效的JavaScript。

插入上述示例代码的`$userInput`中的此输入`0;var date=new Date(); do{curDate = new Date();}while(curDate-date<10000)`将导致执行以下JavaScript函数。这个特定的攻击字符串将使整个MongoDB实例在10秒内以100% CPU使用率执行。

`function() { return obj.credits - obj.debits < 0;var date=new Date(); do{curDate = new Date();}while(curDate-date<10000); }`

### 示例2

即使查询中使用的输入完全清理或参数化，也有另一种可能触发NoSQL注入的路径。许多NoSQL实例有自己的保留变量名，独立于应用程序编程语言。

例如，在MongoDB中，`$where`语法本身是一个保留查询运算符。它需要完全按所示方式传递到查询中；任何更改都会导致数据库错误。但是，因为`$where`也是有效的PHP变量名，攻击者可能通过创建名为`$where`的PHP变量将代码注入查询。PHP MongoDB文档明确警告开发人员：

请确保对所有特殊查询运算符（以`$`开头的）使用单引号，这样PHP就不会尝试用变量`$exists`的值替换`$exists`。

即使查询不依赖于任何用户输入，如以下示例，攻击者也可以用恶意数据替换运算符：

`db.myCollection.find( { $where: function() { return obj.credits - obj.debits < 0; } } );`

一种可能向PHP变量赋值数据的方法是通过HTTP参数污染（参见：[测试HTTP参数污染](04-Testing_for_HTTP_Parameter_Pollution.md)）。通过参数污染创建名为`$where`的变量，可能会触发MongoDB错误，表明查询不再有效。任何`$where`值（除了字符串`$where`本身）都应该足以演示漏洞。攻击者将通过插入以下内容来开发完整漏洞利用：

`$where: function() { //arbitrary JavaScript here }`
