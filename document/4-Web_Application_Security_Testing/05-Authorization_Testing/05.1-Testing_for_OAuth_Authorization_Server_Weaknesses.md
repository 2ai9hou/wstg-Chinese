# 测试 OAuth 授权服务器弱点

## 概述

OAuth 将用户的身份及其相应的访问权限存储在授权服务器（AS）中。AS 在 OAuth 流程中扮演至关重要的角色，因为它授予客户端对资源的访问权限。为了能够安全地做到这一点，它必须正确验证 OAuth 流程中参数的验证。

参数验证失败可能导致账户接管、未授权资源访问和权限提升。

## 测试目标

- 识别授权服务器中的弱点。

## 如何测试

为了测试 AS 弱点，您的目标是：

1. 检索用于授权的凭据。
2. 通过强制浏览为自己授予对任意资源的访问权限。
3. 绕过授权。

### 测试重定向 URI 验证不足

如果 `redirect_uri` 未得到正确验证，可以精心制作一个链接，其中包含指向攻击者控制的服务器的 URL。这可用于欺骗 AS 将授权码发送给攻击者。在以下示例中，`client.evil.com` 被用作伪造的 `redirect_uri`。

```text
https://as.example.com/authorize?client_id=example-client&redirect_uri=http%3A%2F%client.evil.com%2F&state=example&response_mode=fragment&response_type=code&scope=openid&nonce=example
```

如果用户在用户代理中打开此链接，AS 将重定向用户代理到恶意 URL。

攻击者可以捕获伪造 URL 中传递的 `code` 值，然后将其提交到 AS 令牌端点。

以下请求说明了一个授权请求，该请求将 `redirect_uri` 发送到授权服务器。客户端 `client.example.com` 向 AS `as.example.com` 发送授权请求，URL 编码的重定向 URI 为 `http%3A%2F%2Fclient.example.com%2F`。

```http
GET /authorize
    ?redirect_uri=http%3A%2F%2Fclient.example.com%2F
    &client_id=example-client
    &errorPath=%2Ferror
    &scope=openid%20profile%20email
    &response_type=code
    &response_mode=query
    &state=example
    &nonce=example
    &code_challenge=example
    &code_challenge_method=S256 HTTP/1.1
Host: as.example.com
```

AS 使用包含授权码的重定向进行响应。这可以与令牌请求中的访问令牌交换。如下所示，`Location` 头中的 URL 是在上一个 `redirect_uri` 参数中给出的 URI。

```http
HTTP/1.1 302 Found
Date: Mon, 18 Oct 2021 20:46:44 GMT
Content-Type: text/html; charset=utf-8
Content-Length: 340
Location: https://client.example.com/?code=example&state=example
```

要测试 AS 是否容易受到重定向 URI 验证不足的影响，请使用 ZAP 等 HTTP 拦截代理捕获流量。

1. 启动 OAuth 流程并在授权请求处暂停。
2. 更改 `redirect_uri` 的值并观察响应。
3. 调查响应并识别 AS 是否接受了任意的 `redirect_uri` 参数。

如果 AS 将用户代理重定向到您指定的 `redirect_uri`，则 AS 未正确验证 `redirect_uri`。

另外，请参阅[测试服务器端请求伪造](../07-Input_Validation_Testing/19-Testing_for_Server-Side_Request_Forgery.md)中的"常见过滤器绕过"部分，以识别重定向 URI 验证的常见绕过方法。

### 测试授权码注入

在授权码流程代码交换期间，AS 向客户端颁发一个代码，然后与令牌端点交换以检索授权令牌和刷新令牌。

对 AS 执行以下测试：

1. 为另一个 `client_id` 发送有效代码。
2. 为另一个资源所有者发送有效代码。
3. 为另一个 `redirect_uri` 发送有效代码。
4. 多次重发代码（代码重放）。

#### 测试公共客户端

发送到令牌端点的请求包含授权码。它与令牌交换。使用 ZAP 等 HTTP 拦截代理捕获此请求，并使用修改后的值重新发送请求。

```http
POST /oauth/token HTTP/1.1
Host: as.example.com
[...]

{
    "errorPath":"/error",
    "client_id":"example-client",
    "code":"INJECT_CODE_HERE",
    "grant_type":"authorization_code",
    "redirect_uri":"https://client.example.com"
}
```

如果 AS 用 `access_token` 响应，则代码注入成功。

#### 测试机密客户端

由于机密客户端的 OAuth 流程受客户端密钥额外保护，因此不可能直接向令牌端点提交授权码。相反，将授权码注入客户端。这个注入的代码将随客户端密钥一起由机密客户端发送到令牌请求中。

首先，从 AS 捕获授权码：

1. 使用用户 Alice 启动授权码流程。当从 AS 收到代码时暂停。
2. 不要将代码提交给客户端，并记下代码和相应的状态。

然后，注入代码：

1. 使用用户 Mallory 启动授权码流程，并将为用户 Alice 收集的代码和状态值注入到流程中。
2. 当攻击成功时，客户端现在应该拥有一个 `authorization_token`，该令牌授予对用户 Alice 拥有的资源的访问权限。

```http
GET /callback?code=INJECT_CODE_HERE&state=example HTTP/1.1
Host: client.example.com
[...]

```

### 测试 PKCE 降级攻击

在某些情况下，PKCE 扩展可以从授权码流程中移除。这有可能使公共客户端容易受到 PKCE 扩展所减轻的攻击。

当以下情况时会发生：

- AS 不支持 PKCE。
- AS 未正确验证 PKCE。

两者都可以使用 ZAP 等 HTTP 拦截代理进行测试。执行以下测试：

1. 发送不带 `code_challenge=sha256(xyz)` 和 `code_challenge_method` 参数的授权请求。
2. 发送带有空值的 `code_challenge=sha256(xyz)` 参数的授权请求。
3. 发送带有伪造值的 `code_challenge=sha256(xyz)` 参数的授权请求。

以下示例突出显示了要修改的值：

```http
GET /authorize
    ?redirect_uri=http%3A%2F%client.example.com
    &client_id=example-client
    &errorPath=%2Ferror
    &scope=openid%20profile%20email
    &response_type=code
    &response_mode=web_message
    &state=example-state
    &nonce=example-nonce
    &code_challenge=MODIFY_OR_OMIT_THIS
    &code_challenge_method=MODIFY_OR_OMIT_THIS
    &prompt=none HTTP/1.1
Host: as.example.com
[...]

```

AS 应验证令牌交换中的 `code_verifier` 值。要测试：

1. 发送不带 `code_verifier` 的令牌请求。
2. 发送带有空 `code_verifier` 的令牌请求。
3. 发送带有对不同授权码有效 `code_verifier` 的令牌请求。

```http
POST /oauth/token HTTP/1.1
Host: as.example.com
[...]

{
"client_id":"example-client",
"code_verifier":"MODIFY_OR_OMIT_THIS",
"code":"example",
"grant_type":"authorization_code",
"redirect_uri":"https://client.example.com"
}
```

### 测试同意页面跨站请求伪造

CSRF 攻击在 [CSRF](../06-Session_Management_Testing/05-Testing_for_Cross_Site_Request_Forgery.md) 中有描述。OAuth 可能受到 CSRF 攻击。

为了防止 CSRF 攻击，OAuth 利用 `state` 参数作为反 CSRF 令牌。

其他措施也可以防止 CSRF 攻击。PKCE 流程可以减轻 CSRF。`nonce` 值也可以作为反 CSRF 令牌。

根据 [CSRF](../06-Session_Management_Testing/05-Testing_for_Cross_Site_Request_Forgery.md) 测试用例中描述的测试，测试 OAuth 使用的每个反 CSRF 参数的每个请求。

同意页面显示给用户，以验证该用户同意客户端代表用户访问资源。用 CSRF 攻击同意页面可能授予任意客户端代表用户访问资源的权限。此流程的步骤如下：

1. 客户端生成状态参数并将其与同意请求一起发送。
2. 用户代理显示同意页面。
3. 资源所有者授予客户端访问权限。
4. 同意与确认的范围一起发送到 AS。

使用 ZAP 等 HTTP 拦截代理测试 `state` 参数是否得到正确验证。

```http
POST /u/consent?state=Tampered_State HTTP/1.1
Host: as.example.com
[...]

state=MODIFY_OR_OMIT_THIS
&audience=https%3A%2F%2Fas.example.com%2Fuserinfo
&scope%5B%5D=profile
&scope%5B%5D=email
&action=accept
```

### 测试点击劫持

点击劫持在 [测试点击劫持](../11-Client-side_Testing/09-Testing_for_Clickjacking.md) 中有描述。当同意页面容易受到点击劫持且攻击者拥有 `client_id`（对于公共客户端）或客户端密钥（对于机密客户端）时，攻击者可以伪造用户的同意并通过恶意客户端获得对请求资源的访问权限。

#### 如何测试

为了使此攻击成功，攻击者需要在一个 iframe 中加载授权页面。

以下 HTML 页面可用于在 iframe 中加载授权页面：

```html
<html>
    <head>
        <title>点击劫持测试页面</title>
    </head>
    <body>
        <iframe src="https://as.example.com/auth/realms/example/login-actions/required-action?execution=OAUTH_GRANT&client_id=example-client" width="500" height="500"></iframe>
    </body>
</html>
```

如果成功加载，则该站点容易受到点击劫持攻击。

有关如何进行此类攻击的详细描述，请参阅[测试点击劫持](../11-Client-side_Testing/09-Testing_for_Clickjacking.md)。

### 测试令牌生命周期

OAuth 有两种类型的令牌：访问令牌和刷新令牌。访问令牌的有效期应该有限。这意味着它是短命的：好的持续时间取决于应用程序，可能是 5 到 15 分钟。

刷新令牌的有效期应该更长。它应该是一次性令牌，每次使用后都会被替换。

#### 测试访问令牌生命周期验证

当 JSON Web Token（JWT）被用作访问令牌时，可以从解码的 JWT 中获取访问令牌的有效性。这在[测试 JSON Web Token](../06-Session_Management_Testing/10-Testing_JSON_Web_Tokens.md) 中有描述。AS 可能无法正确验证 JWT 的生命周期。

要测试访问令牌的生命周期，请使用 ZAP 等 HTTP 拦截代理。拦截包含访问令牌的端点请求。将此请求放入重放器中，让目标时间过去。访问令牌的有效期应在 5 到 15 分钟之间，具体取决于资源的敏感性。

此类请求可能类似于以下示例。令牌也可以通过其他方式传输，例如在 cookie 中。

```http
GET /userinfo HTTP/1.1
Host: as.example.com
[...]
Authorization: Bearer eyJhbGciOiJkaXIiL[...]

```

通过在不同时间长度后发送请求来测试生命周期验证，例如 5 分钟、10 分钟和 30 分钟后。

可以通过自动化步骤和服务器响应日志来优化此过程。当收到 HTTP 状态 403（而不是 HTTP 状态 200）时，可以表明访问令牌不再有效。

#### 测试刷新令牌生命周期验证

刷新令牌的有效期比访问令牌长。由于其较长的有效期，刷新令牌在交换访问令牌后应被失效。

刷新令牌在与访问令牌一起发放给客户端的同一令牌请求中发放。

使用 ZAP 等 HTTP 拦截代理。按以下方式设置测试：

1. 检索有效的刷新令牌。
2. 捕获用于将刷新令牌交换为新访问令牌的请求。
3. 将捕获的请求发送到请求重放器。

在以下示例中，刷新令牌作为 POST 主体的一部分发送。

```http
POST /token HTTP/1.1
Host: as.example.com
Cookie: [...]
[...]

grant_type=refresh_token
&refresh_token=eyJhbGciOiJIUz[...]
&client_id=example-client

```

执行以下测试：

1. 发送刷新令牌并确定 AS 是否发出访问令牌。
2. 使用相同的刷新令牌重复步骤，以评估单个刷新令牌被接受的次数。

当 JWT 被用作刷新令牌时，可以从解码的 JWT 中获取刷新令牌的有效性。这在[测试 JSON Web Token](../06-Session_Management_Testing/10-Testing_JSON_Web_Tokens.md) 中有描述。刷新令牌的有效期可能更长，但应有到期日期。

额外的安全性可以通过被盗检测机制获得。如果刷新令牌在其有效性（或生命周期）之外被用于令牌交换，AS 将使所有刷新令牌失效。要测试此机制：

1. 发送刷新令牌并确定 AS 是否发出访问令牌。
2. 使用相同的刷新令牌重复步骤，直到它失效。
3. 使用最后一个令牌响应中的刷新令牌。

如果发放给此资源所有者客户端的所有刷新令牌都被失效，则 AS 具有令牌被盗检测功能。

## 修复方案

大多数针对 OAuth AS 的攻击都可以通过在授权码和令牌交换期间验证参数的存在和内容来缓解。

限制凭据（如授权码和刷新令牌）的时间跨度和允许使用次数。这可以减轻某些类型的攻击，如果这些凭据被获取，也会限制攻击者对它们的使用。

正确配置安全缓解措施（如 CORS、反 CSRF 令牌和反点击劫持头）可以减轻或限制攻击的影响。

- 始终验证所有参数是否存在，并验证其值。
- 使用 PKCE 扩展来正确保护授权码和令牌交换。
- 不允许安全功能（如 PKCE 扩展）回退。
- 限制凭据的生命周期。
- 尽可能仅使用一次凭据，例如授权码。
- 配置可用的安全缓解措施，如 CORS、反 CSRF 令牌和反点击劫持头。
