# 测试 OAuth 客户端弱点

## 概述

OAuth 授予客户端对资源的访问权限。这允许他们代表资源所有者行事。客户端在令牌交换中接收授权码和刷新令牌并存储它们。

保护令牌交换和凭据失败可能导致未授权资源访问和权限提升。

## 测试目标

- 识别 OAuth 客户端中的弱点。

## 如何测试

为了测试客户端弱点，您的目标是：

1. 检索用于授权的凭据。
2. 通过强制浏览为自己授予对任意资源的访问权限。
3. 绕过授权。

### 测试暴露的客户端密钥

客户端密钥用于在客户端向授权服务器（AS）进行身份验证时证明客户端是可信来源。

公共客户端通常无法安全存储客户端密钥。

要识别客户端侧代码中的客户端密钥，对客户端侧代码进行侦察。

1. 浏览到应用程序。
2. 打开浏览器的开发者工具。
3. 导航到调试器选项卡。
4. 按 Ctrl+Shift+F 打开搜索。
5. 搜索类似于 `client-secret` 的术语，并确定是否找到任何内容。

如果不成功，您也可以：

1. 使用 ZAP 等 HTTP 拦截代理逐步完成授权过程。
2. 从 `client-secret` 参数的 URI 中检索客户端密钥。
3. 用客户端密钥的值替换上述搜索词，并确定它是否被暴露。

### 测试不正确的令牌存储

客户端接收访问令牌，并理想地存储在可以保护这些令牌免受攻击者攻击的位置。

机密客户端应将令牌存储在易失性内存中，以防止通过其他攻击（如本地文件包含、能够访问环境攻击者或 SQL 注入攻击）进行访问。

公共客户端（如单页应用程序）无法安全地存储令牌。例如，跨站脚本攻击允许攻击者访问存储在浏览器中的凭据。

公共客户端可以将令牌存储在浏览器会话存储或 cookie 中，但不能存储在本地存储中。要确定令牌是否存储不当：

1. 浏览到应用程序。
2. 检索访问令牌。
3. 打开浏览器的开发者工具。
4. 导航到应用程序选项卡。
5. 找到本地存储并查看存储的数据。
6. 找到会话存储并查看存储的数据。
7. 找到 Cookie 存储并查看存储的数据。

### 测试访问令牌注入

仅当客户端使用直接向客户端颁发访问令牌的响应类型时，此攻击才可能发生。这发生在隐式流、资源所有者密码凭据和机器对机器流程的授予类型中。请参阅[测试 OAuth 弱点](05-Testing_for_OAuth_Weaknesses.md)获取进一步描述。

当访问令牌泄露给攻击者，然后用于向合法客户端进行身份验证时，访问令牌注入成功。

要测试访问令牌注入，请按照以下步骤操作。在此示例中，授权令牌（`ZXhhbXBsZQo=`）已被泄露。

1. 拦截客户端和授权服务器之间的流量。
2. 使用隐式流授予类型启动客户端的 OAuth 流程。
3. 注入被盗的访问令牌：
    - 使用被盗的访问令牌（`ZXhhbXBsZQo=`）向客户端发送伪造的授权响应。
    - 拦截有效授权响应并用泄露的令牌（`ZXhhbXBsZQo=`）替换访问令牌（`dGVzdGluZwo=`）。

## 修复方案

- 仅在客户端有能力安全存储的情况下使用客户端密钥。
- 遵循安全存储令牌的最佳实践。对它们给予与其他凭据相同的安全考虑。
- 避免使用已弃用的 OAuth 授予类型。请参阅[测试 OAuth 弱点](05-Testing_for_OAuth_Weaknesses.md)获取进一步描述。
