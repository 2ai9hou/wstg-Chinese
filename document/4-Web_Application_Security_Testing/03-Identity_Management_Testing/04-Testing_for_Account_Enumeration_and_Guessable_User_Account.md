# 测试账户枚举和可猜测的用户账户

|ID          |
|------------|
|WSTG-IDNT-04|

## 概述

此测试的范围是验证是否可以通过与应用程序的身份验证机制交互来收集一组有效的用户名。此测试对暴力破解测试很有用，在暴力破解测试中，测试人员验证给定有效用户名后，是否有可能找到相应的密码。

通常，Web 应用程序会在用户名存在于系统上时泄露信息，这可能是由于错误配置或设计决策造成的。例如，有时当我们提交错误的凭据时，会收到一条消息，说明用户名存在于系统上或提供的密码错误。攻击者可以使用获取的信息来获取系统上的用户列表。这些信息可用于攻击 Web 应用程序，例如通过暴力破解或默认用户名和密码攻击。

测试人员应与应用程序的身份验证机制进行交互，以了解发送特定请求是否会导致应用程序以不同方式回答。此问题的存在是因为当用户提供有效用户名时，Web 应用程序或 Web 服务器发布的信息与使用无效用户名时的信息不同。

在某些情况下，会收到一条消息，揭示提供的凭据是否错误，因为使用了无效用户名或无效密码。有时，测试人员可以通过发送用户名和空密码来枚举现有用户。

## 测试目标

- 审查与用户识别相关的流程（如注册、登录等）。
- 通过响应分析尽可能枚举用户。

## 如何测试

在黑盒测试中，测试人员对特定应用程序、用户名、应用程序逻辑、登录页面上的错误消息或密码恢复功能一无所知。如果应用程序存在漏洞，测试人员会收到一条响应消息，直接或间接地揭示一些对枚举用户有用的信息。

### HTTP 响应消息

#### 测试有效凭据

记录提交有效用户 ID 和有效密码时服务器的答复。

> 使用 Web 代理，注意从这次成功的身份验证中获取的信息（HTTP 200 响应，响应长度）。

#### 测试有效用户但密码错误

现在，测试人员应尝试输入有效用户 ID 和错误密码，并记录应用程序生成的错误消息。

> 浏览器应显示类似于以下的消息：
>
> ![认证失败](images/AuthenticationFailed.png)\
> *图 4.3.4-1：认证失败*
>
> 与任何揭示用户存在性的消息不同，例如：
>
> `Login for User foo: invalid password`
>
> 使用 Web 代理，注意从这次不成功的身份验证尝试中获取的信息（HTTP 200 响应，响应长度）。

#### 测试不存在的用户名

现在，测试人员应尝试输入无效用户 ID 和错误密码，并记录服务器答复（测试人员应确信该用户名在应用程序中无效）。记录错误消息和服务器答复。

> 如果测试人员输入不存在的用户 ID，他们会收到类似以下的消息：
>
> ![此用户未激活](images/Userisnotactive.png)\
> *图 4.3.4-3：此用户未激活*
>
> 或类似以下的消息：
>
> `Login failed for User foo: invalid Account`
>
> 通常，应用程序应对不同的错误请求做出相同的错误消息和长度响应。如果响应不相同，测试人员应进行调查并找出在两个响应之间产生差异的关键。例如：
>
> 1. 客户端请求：有效用户/错误密码
> 2. 服务器响应：密码不正确
> 3. 客户端请求：错误用户/错误密码
> 4. 服务器响应：用户未识别
>
> 上述响应让客户端了解对于第一个请求他们有一个有效的用户名。因此他们可以与应用程序交互，请求一组可能的用户 ID 并观察答复。
>
> 查看第二个服务器响应，测试人员同样可以了解他们没有一个有效的用户名。因此他们可以以相同的方式进行交互，并通过观察服务器响应来创建有效用户 ID 列表。

### 枚举用户的其他方法

测试人员可以通过多种方式枚举用户，例如：

#### 分析登录页面收到的错误代码

一些 Web 应用程序发布可分析的具体错误代码或消息。

#### 分析 URL 和 URL 重定向

例如：

- `https://www.foo.com/err.jsp?User=baduser&Error=0`
- `https://www.foo.com/err.jsp?User=gooduser&Error=2`

如上所示，当测试人员向 Web 应用程序提供用户 ID 和密码时，他们会在 URL 中看到指示发生错误的消息。在第一种情况下，他们提供了错误的用户 ID 和错误密码。在第二种情况下，正确的用户 ID 和错误密码，因此他们可以识别有效的用户 ID。

#### URI 探测

有时，Web 服务器在接收到对已存在目录或不存在的目录的请求时会有不同的响应。例如在一些门户中，每个用户都与一个目录相关联。如果测试人员尝试访问已存在的目录，他们可能会收到 Web 服务器错误。

Web 服务器收到的常见错误包括：

- 403 禁止访问错误代码
- 404 未找到错误代码

示例：

- `https://www.foo.com/account1` - 我们从 Web 服务器收到：403 禁止访问
- `https://www.foo.com/account2` - 我们从 Web 服务器收到：404 文件未找到

在第一种情况下，用户存在，但测试人员无法查看网页；在第二种情况下，用户"account2"不存在。通过收集这些信息，测试人员可以枚举用户。

#### 分析网页标题

测试人员可以从网页标题获取有用信息，获得可揭示问题出在用户名还是密码的具体错误代码或消息。

例如，如果用户无法对应用程序进行身份验证并收到标题类似于以下内容的网页：

- `Invalid user`
- `Invalid authentication`

#### 分析恢复功能收到的消息

当我们使用恢复功能（即忘记密码功能）时，漏洞应用程序可能会返回揭示用户名是否存在的消息。

例如，类似于以下的消息：

- `Invalid username: email address is not valid or the specified user was not found.`
- `Valid username: Your password has been successfully sent to the email address you registered with.`

#### 友好的 404 错误消息

当我们请求不存在的目录中的用户时，并不总是收到 404 错误代码。相反，我们可能会收到带有图片的"200 OK"，在这种情况下，我们可以假设当我们收到特定图片时，用户不存在。这种逻辑可以应用于其他 Web 服务器响应；诀窍是对 Web 服务器和 Web 应用程序消息进行良好分析。

#### 分析响应时间

除了查看响应的内容外，还应考虑响应所花费的时间。特别是在请求导致与外部服务交互时（如发送忘记密码的电子邮件），这可能为响应增加数百毫秒，这可用于确定请求的用户是否有效。

### 猜测用户

#### 可预测的用户名结构

在许多组织中，用户名遵循一致的格式（如名字首字母加姓氏，如 jbloggs，或结构化标识符如 CN000100）。如果可以识别命名约定，通常可以系统地导出有效用户名。

测试人员应：

- 识别用户名是否遵循可预测的结构。
- 尝试根据观察到的模式生成其他用户名。
- 使用从组织数据派生的用户名词典（如员工姓名、电子邮件格式）。
- 观察应用程序响应以确定生成的用户名是否有效。

可预测的用户名结构大大减少了枚举有效账户所需的工作量，并可能促进暴力破解等进一步攻击。

在某些情况下，用户 ID 是根据管理员或公司的特定策略创建的。例如，我们可以看到按顺序创建的用户 ID：

```text
CN000100
CN000101
...
```

有时用户名是用 REALM 别名和连续数字创建的：

- R1001 – REALM1 的用户 001
- R2001 – REALM2 的用户 001

在上述示例中，我们可以创建简单的 shell 脚本，组成用户 ID 并使用 wget 等工具提交请求，以自动执行 Web 查询来辨别有效的用户 ID。我们还可以使用 Perl 和 cURL 创建脚本。

其他可能性包括：- 与信用卡号码关联的用户 ID，或一般有规律的数字。- 与真实姓名关联的用户 ID，例如，如果 Freddie Mercury 的用户 ID 是"fmercury"，则您可能猜测 Roger Taylor 的用户 ID 是"rtaylor"。

同样，我们可以从 LDAP 查询或 Google 信息收集中获取的信息猜测用户名，例如从特定域。Google 可以通过特定查询或简单的 shell 脚本或工具帮助找到域用户。

> 通过枚举用户账户，您可能在使用预定数量的失败探测后锁定账户（基于应用程序策略）。此外，有时您的 IP 地址可能会被应用程序防火墙或入侵防御系统上的动态规则禁止。

### 测试员工冒充

确保未注册用户无法在注册过程中选择保留的用户名（如 admin、administrator、moderator）。此外，验证用户无法在个人资料编辑页面上将其当前用户名更改为这些保留用户名之一。

如果 Web 应用程序具有允许用户访问 Web 应用程序的注册和个人资料编辑功能，则要测试的交互包括：

- 注册流程：
    - 以未注册用户身份访问注册页面并填写注册表单，输入保留的用户名之一（如 admin、administrator、moderator），提交注册表单，然后验证响应。
    - 注册流程应拒绝表单提交并显示错误消息，表明所选用户名不可用于注册。
- 个人资料编辑页面：
    - 使用有效凭据登录 Web 应用程序并导航到个人资料编辑页面。尝试将当前用户名更改为保留用户名之一（如 admin、administrator、moderator）并保存更改以验证行为。
    - 个人资料编辑流程应拒绝用户名更改请求并显示错误消息，表明所选用户名不可用。
- 测试变体和相似形式：
    - 对保留用户名（如 Admin、ADMIN、Administrator）的不同变体重复上述步骤，并使用不同的大小写字母组合进行测试，以确保正确处理大小写不敏感。
    - Web 应用程序应将这些变体视为与保留用户名相同，拒绝其选择或修改。

### 灰盒测试

#### 测试身份验证错误消息

验证应用程序对每个产生失败身份验证的客户端请求以相同方式回答。对于此问题，黑盒测试和灰盒测试具有相同的概念，基于从 Web 应用程序收到的消息或错误代码的分析。

> 应用程序应对每次失败的认证尝试以相同方式回答。
>
> 例如：*提交的凭据无效*

## 修复建议

确保应用程序在登录过程中输入无效账户名、密码或其他用户凭据时返回一致的一般错误消息。

确保默认系统账户和测试账户在将系统投入生产之前（或将其暴露于不受信任的网络之前）被删除。
